## Skeptical Triage — Candidatos NOT-REAL

### A-07 — `deile/config/manager.py:662` — get_event_loop em setup_hot_reload

A chamada `asyncio.get_event_loop()` em linha 662 ocorre dentro de uma `async def` (confirmado: `setup_hot_reload` é um método assíncrono). Dentro de um contexto assíncrono em Python 3.10+, `get_event_loop()` retorna o loop em execução sem emitir `DeprecationWarning` — o warning só ocorre quando chamado fora de qualquer loop. O DEILE roda com um único loop persistente (uvicorn ou `asyncio.run()` no K8s worker). O loop capturado é válido pelo tempo de vida do processo. Usar `asyncio.get_running_loop()` seria mais idiomático, mas a mudança é cosmética.

### B6 — `deile/events/event_bus.py:389-398` — eviction manual redundante

Verificado experimentalmente: `d[0]` é o elemento a ser evictado tanto pelo `discard(evicted)` manual quanto pelo `d.append()` com `maxlen`. Ambas as operações targetam o mesmo objeto. O repro fornecido (`deque(maxlen=3)`) confirma que `s == set(d)` se mantém após N inserções. Código correto, embora verbose.

### B7 — `deile/events/event_bus.py:422-426` — list.pop(0) O(n)

Com `n = 1000` (constante hard-coded), `list.pop(0)` realiza 1000 shifts com constante de tempo minúscula. Não é bug de corretude. Usar `collections.deque(maxlen=1000)` seria melhoria cosmética de O(1) vs O(n), mas o impacto prático é irrelevante nessa escala.

### B8 — `deile/orchestration/pipeline/scheduler.py:296-308` — re-execução pós-crash

O docstring de `ScheduleStore` em linha 29 documenta "catch-up coalesces by default". Todos os handlers de stage usam `transition_issue` e `add_labels` que são no-ops se o label já está no estado target. Re-execução após crash é semanticamente segura por design.

### B9 — `deile/orchestration/pipeline/lockfile.py:37-45` — _pid_alive errno

`os.kill(pid, 0)` retorna sem erro se o processo existe e temos permissão; lança `OSError(ESRCH)` se não existe; lança `OSError(EPERM)` se existe mas sem permissão. `exc.errno != errno.ESRCH` corretamente retorna `True` para EPERM (processo vivo) e `False` para ESRCH (processo morto). Signal 0 nunca gera `EINVAL`. Lógica POSIX correta.

### C-005 — `deile/tools/worktree_tool.py:47-60` — symlink TOCTOU

`CLAUDE.md` do projeto confirma `readOnlyRootFilesystem: true` para o pod K8s. Nenhum ator externo pode criar symlinks dentro de `.worktrees/` porque o filesystem é read-only. Em máquinas de desenvolvimento, o diretório pertence ao usuário que executa o DEILE — que já tem permissão de deleção por outros meios. O vetor de ataque requer um ator concorrente privilegiado que não existe no deployment.

### C-006 — `deile/runtime/instance_state.py:506-514` — get_event_loop em atexit

A linha 508 captura `RuntimeError` explicitamente (`except RuntimeError: loop = None`). Quando `loop is None`, o código cai no branch de unlink do socket Unix (linhas 522-525), que é o comportamento correto de limpeza best-effort para o path de atexit (sem loop). O `DeprecationWarning` em Python 3.10-3.11 é harmless (warning apenas). Sem falha funcional.

### C-007 — `deile/skills/watcher.py:112-116` — join com timeout

`_stop_event.set()` é chamado em linha 114, antes do `join(timeout=2.0)`. O método `run()` verifica `_stop_event.is_set()` no loop externo (linha 119) e interno (linha 138). A linha 145 tem `if not self._stop_event.is_set(): self._trigger()` impedindo novos triggers após stop. No pior caso, uma `_trigger_reload()` em andamento termina após o join — `registry.replace_all()` é atômico, sem corrupção de estado.

### C-008 — `deile/skills/registry.py:31-39` — log fora do lock

`previous` é uma variável local Python capturada dentro do lock (linha 33). `skill` é o parâmetro da função (imutável neste escopo). Nenhuma dessas referências é relida do dict. O GC não coleta `previous` enquanto a função mantém referência. O log descreve com precisão o evento que acabou de ocorrer, mesmo que um evento subsequente já tenha ocorrido. Não afeta corretude do registry.

### C-009 — `deile/runtime/registry.py:284-315` — inode de lock file

O lock file é criado pelo próprio código do registry e nunca é deletado explicitamente (grep confirmado: nenhuma chamada a `unlink()` no lock file em todo o codebase). No K8s, pods únicos por papel nunca compartilham o mesmo process space. Em testes, arquivos estão em `tmp_path` único por teste. O cenário teórico de inode diferente requer um `rm` externo concorrente que não existe no deployment.

### C-010 — `deile/runtime/instance_state.py:431-438` — casts int/float

`_state` é inicializado em `__init__` (linhas 196-203) com literais `int`/`float` e é apenas escrito em disco (nunca deserializado de volta para o mesmo objeto `InstanceState`). ### D-006 — `deile/security/permissions.py:55-59` — re.match vs re.fullmatch

Todas as 8 regras default em `_load_default_rules()` terminam com `.*` ou `$`. O problema só manifesta com regras custom mal escritas sem âncora terminal. ### D-007 — `deile/security/permissions.py:435-440` — singleton sem lock

O GIL do CPython torna check-create-assign efetivamente atômico para operações simples em dict. O DEILE usa arquitetura asyncio de single event loop por processo — não há threads reais chamando `get_permission_manager()` concorrentemente. Ambas as instâncias hipotéticas carregariam o mesmo YAML, produzindo estado idêntico.

### E-006 — `deile/integrations/bot/client.py:158` — exceções não capturadas

O docstring em `client.py:132-135` documenta explicitamente: "Non-transient errors (auth, not-ready, upstream 4xx) fail immediately". `BotClientAuthError` e `BotClientNotReady` não são capturados pelo `_retry_messaging` por design. Propagação é comportamento correto e documentado.

### A-01 — `deile/core/session_store.py:56-64`

**Categoria**: concurrency | **Severidade**: Major

**Hipótese**: `SessionStore.init()` não é protegido por `self._lock`. Dois callers concorrentes passam o guard `if self._db is not None: return` em linha 57 antes que qualquer um tenha feito o assignment de `self._db`, pois o assignment em linha 60 só ocorre após o ponto de suspensão `await`. Ambos chamam `aiosqlite.connect()`, executam PRAGMAs e `executescript()`. O segundo assignment sobrescreve `self._db`, vazando a primeira conexão aberta — que nunca é fechada.

**Repro mínimo**:
```python
import asyncio
from deile.core.session_store import SessionStore

async def demo(tmp_path):
    store = SessionStore(tmp_path / 'x.db')
    await asyncio.gather(store.init(), store.init())
    # Primeira conexão vazada; self._db é a segunda

asyncio.run(demo('/tmp'))
```

**Saída do repro confirmada**:
```
Results: [OperationalError('database is locked'), None]
Total connect() calls: 2
conn[0] id=281472937431696: _running=True  (LEAKED)
conn[1] id=281472937429456: _running=True  (kept as store._db)
Potential leaked connections: 1
```

**Comportamento esperado**: `init()` deve ser idempotente e thread-safe; apenas uma conexão deve ser criada mesmo com chamadas concorrentes.

**Fix proposto**: Double-checked locking com `self._lock` existente — guard externo como fast path, re-check interno sob o lock:
```python
async def init(self) -> None:
    if self._db is not None:
        return
    async with self._lock:
        if self._db is not None:
            return
        self.db_path.parent.mkdir(parents=True, exist_ok=True)
        self._db = await aiosqlite.connect(self.db_path)
        await self._db.execute("PRAGMA journal_mode=WAL;")
        await self._db.execute("PRAGMA synchronous=NORMAL;")
        await self._db.executescript(_SCHEMA)
        await self._db.commit()
```

**Testes impactados**: `deile/tests/core/test_session_store.py` (adicionar test de init concorrente), `deile/tests/core/test_bot_hooks_e2e.py`

**Docs impactados**: Nenhum

### A-02 — `deile/core/session_store.py:136`

**Categoria**: correctness | **Severidade**: Minor

**Hipótese**: `purge_older_than()` usa `datetime.fromtimestamp(cutoff, tz=timezone.utc).isoformat()` que produz sufixo `+00:00`. Os timestamps armazenados por `_utc_iso()` em linha 36 usam `strftime('%Y-%m-%dT%H:%M:%S.%fZ')` com sufixo `Z`. SQLite compara lexicograficamente. No boundary exato com microsegundos zero, `'2026-06-18T12:00:00.000000Z' < '2026-06-18T12:00:00+00:00'` avalia como `False` porque `'.'` (ASCII 46) > `'+'` (ASCII 43) na posição 19.

**Repro mínimo**:
```python
from datetime import datetime, timezone
one_day_ago = datetime(2026, 6, 18, 12, 0, 0, 0, tzinfo=timezone.utc)
stored = one_day_ago.strftime('%Y-%m-%dT%H:%M:%S.%fZ')
cutoff = datetime.fromtimestamp(one_day_ago.timestamp(), tz=timezone.utc).isoformat()
assert stored < cutoff  # FALHA: False, não True
```

**Saída do repro confirmada**:
```
stored: 2026-06-18T12:00:00.000000Z
cutoff: 2026-06-18T12:00:00+00:00
stored < cutoff: False
```

**Fix proposto**: Linha 136 — substituir `.isoformat()` por `.strftime('%Y-%m-%dT%H:%M:%S.%fZ')` para consistência com `_utc_iso()`, `count_sessions_before()` (linha 172) e `delete_sessions_before()` (linha 185).

**Testes impactados**: `deile/tests/core/test_session_store.py` (`test_purge_zero_clears` — verificar deleção exata no boundary)

**Docs impactados**: Nenhum

### A-03 — `deile/config/manager.py:748-758`

### A-04 — `deile/config/manager.py:256-263`

**Categoria**: state-leak | **Severidade**: Major

**Hipótese**: `_save_yaml()` linha 262 usa `open(file_path, 'w', ...)` que trunca o arquivo antes de escrever. Crash, OOM kill ou disk-full durante a escrita deixa o YAML vazio/corrompido. Na próxima inicialização, `_load_yaml()` retorna `None` e o ConfigManager cai para defaults — silenciosamente perdendo todos os comandos, configurações e preferências do usuário. `env_store._save_raw` (linhas 55-80) faz corretamente com tmpfile + fsync + `os.replace`.

**Repro mínimo**:
```python
f = open('/tmp/config.yaml', 'w')
# Arquivo truncado. Se processo morrer aqui:
import yaml; yaml.safe_load(open('/tmp/config.yaml'))  # retorna None
```

**Saída do repro confirmada**:
```
Arquivo vazio (truncado em open) → yaml.safe_load retorna None
YAML parcialmente corrompido → ParserError/ScannerError
Em ambos os casos: silent config loss, fallback para defaults
```

**Fix proposto**: Substituir `_save_yaml()` pelo padrão atomic de `env_store._save_raw`: escrever em `NamedTemporaryFile` no mesmo diretório, `flush()` + `os.fsync()`, `os.replace(tmp, dest)`.

**Testes impactados**: `deile/tests/config/test_config_manager_thread_safety.py`

### A-05 — `deile/core/loop_guard.py:200-209`

**Categoria**: state-leak | **Severidade**: Minor

**Hipótese**: `ToolLoopGuard.__post_init__` em linha 200 compara settings com literais hard-coded (`if s.loop_guard_max_calls != 50:`). Se settings têm valor não-default (e.g. 30), o override aplica mesmo quando o construtor recebeu argumento explícito (e.g. `max_calls=2`). O argumento do construtor é silenciosamente ignorado. Afeta principalmente testes que constroem `ToolLoopGuard(max_calls=N)` com settings monkeypatched.

**Repro mínimo**:
```python
from unittest.mock import patch
from deile.core.loop_guard import ToolLoopGuard
with patch('deile.config.settings.get_settings') as mock:
    mock.return_value.loop_guard_max_calls = 30
    mock.return_value.loop_guard_disabled = False
    # ... outros campos
    g = ToolLoopGuard(max_calls=2)
    assert g.max_calls == 2  # FALHA: g.max_calls é 30
```

**Saída do repro confirmada**:
```
g.max_calls = 30
Expected: 2
Bug triggered: True
```

**Fix proposto**: Comparar settings com o default do campo, não com literal hard-coded. Aplicar override apenas quando o campo ainda está no default (caller não passou argumento explícito):
```python
if s.loop_guard_max_calls != defaults["max_calls"] and self.max_calls == defaults["max_calls"]:
    self.max_calls = max(1, s.loop_guard_max_calls)
```

**Testes impactados**: `deile/tests/test_loop_detection.py`
